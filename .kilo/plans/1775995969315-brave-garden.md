## Implementation Notes

1. **Status values**: Each service table should have a status column for tracking:
   - `vaccination_records.status`: pending → approved → completed
   - `kapon_records.status`: pending → approved → completed / rejected
   - `adoption_applications.status`: pending → approved → rejected

2. **Pet selection limits** (as per form UI):
   - Vaccination: Max 3 pets per transaction
   - Kapon: Max 3 pets per transaction  
   - Adoption: Max 2 pets per application

3. **Existing tables used**:
   - `users` - already exists
   - `pet_owners` - already exists (with alternate_phone_number)
   - `pets` - already exists (user's registered pets)
   - `adoption_pets` - already exists (shelter pets available for adoption)

4. **Pet Selection Sources**:
   - **Vaccination & Kapon**: Users select from their **own registered pets** (from `pets` table)
   - **Adoption**: Users select from **shelter pets available for adoption** (from `adoption_pets` table)

5. **Relationship Flow**:
   ```
   users → pet_owners → pets (user's own pets)
   
   vaccination_records → vaccination_pets → pets
   
   kapon_records → kapon_pets → pets
                    → kapon_pet_photos
                    → kapon_pet_agreements
                    → kapon_general_agreements
   
   adoption_applications → adoption_selected_pets → adoption_pets (shelter)
                          → adoption_alternate_contacts
                          → adoption_questionnaires
                          → adoption_home_photos
                          → adoption_valid_ids
                          → adoption_interview_schedules
   ```
