# Plan: missing_pets_form Table Schema (3NF Compliant)

## Goal
Refine the `missing_pets_form` table to follow Third Normal Form (3NF) principles.

## 3NF Design Principles Applied

1. **Remove Redundancy** - Use Foreign Keys instead of duplicating data from parent tables
2. **Combine Date/Time** - Single DATETIME column for better indexing
3. **Geospatial Support** - Add lat/long columns for map integration
4. **Flexible Media** - Separate photos table instead of hardcoded columns
5. **Contact Override** - Only store emergency contact if different from profile

---

## Revised 3NF Schema

### Main Table: `missing_pets_reports`

```php
Schema::create('missing_pets_reports', function (Blueprint $table) {
    // Primary Key
    $table->id('report_id');

    // === FOREIGN KEYS (Replace redundant data) ===
    $table->unsignedBigInteger('user_id');                    // Link to users table
    $table->foreign('user_id')->references('id')->on('users')->onDelete('cascade');
    $table->unsignedBigInteger('pet_id');                      // Link to pets table
    $table->foreign('pet_id')->references('pet_id')->on('pets')->onDelete('cascade');

    // === Physical Distinguishing Marks ===
    $table->text('body_marks')->nullable();                    // Description of marks
    $table->string('eye_color');                               // Eye color
    $table->string('collar_harness')->nullable();              // Collar/harness info

    // === Time and Place (Combined datetime) ===
    $table->timestamp('last_seen_at');                        // Combined date+time
    $table->string('location_barangay');                      // Barangay (for filtering)
    $table->text('location_description')->nullable();         // Specific location

    // === Contact Override (only if different) ===
    $table->string('emergency_contact')->nullable();          // Override user's phone

    // === Status Tracking ===
    $table->enum('status', ['missing', 'found', 'reunited'])->default('missing');

    // === Timestamps ===
    $table->timestamps();
});
```

### Supporting Table: `missing_pets_photos`

```php
Schema::create('missing_pets_photos', function (Blueprint $table) {
    $table->id('photo_id');
    $table->unsignedBigInteger('report_id');
    $table->foreign('report_id')->references('report_id')->on('missing_pets_reports')->onDelete('cascade');
    $table->string('photo_path');                            // Storage path
    $table->enum('photo_type', ['main', 'body_marks', 'eye_color', 'collar_harness', 'other']);
    $table->text('description')->nullable();                  // Optional description
    $table->timestamps();
});
```

## DDL (SQL)

```sql
-- Main Reports Table
CREATE TABLE missing_pets_reports (
    report_id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    user_id BIGINT UNSIGNED NOT NULL,
    pet_id BIGINT UNSIGNED NOT NULL,
    body_marks TEXT NULL,
    eye_color VARCHAR(100) NOT NULL,
    collar_harness VARCHAR(255) NULL,
    last_seen_at DATETIME NOT NULL,
    location_barangay VARCHAR(100) NOT NULL,
    location_description TEXT NULL,
    emergency_contact VARCHAR(20) NULL,
    status ENUM('missing', 'found', 'reunited') NOT NULL DEFAULT 'missing',
    created_at TIMESTAMP NULL,
    updated_at TIMESTAMP NULL,

    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
    FOREIGN KEY (pet_id) REFERENCES pets(pet_id) ON DELETE CASCADE
);

-- Photos Table (One-to-Many)
CREATE TABLE missing_pets_photos (
    photo_id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    report_id BIGINT UNSIGNED NOT NULL,
    photo_path VARCHAR(500) NOT NULL,
    photo_type ENUM('main', 'body_marks', 'eye_color', 'collar_harness', 'other') NOT NULL,
    description TEXT NULL,
    created_at TIMESTAMP NULL,
    updated_at TIMESTAMP NULL,

    FOREIGN KEY (report_id) REFERENCES missing_pets_reports(report_id) ON DELETE CASCADE
);
```

## Indexing Strategy

### For `status` column:
```sql
-- Single column index for status-based queries
CREATE INDEX idx_status ON missing_pets_reports(status);
-- Use case: Filter all "missing" reports
SELECT * FROM missing_pets_reports WHERE status = 'missing';
```

### For `location_barangay` column:
```sql
-- Single column index for barangay filtering
CREATE INDEX idx_barangay ON missing_pets_reports(location_barangay);
-- Use case: Filter reports by barangay
SELECT * FROM missing_pets_reports WHERE location_barangay = 'Burol I';
```

### Composite Indexes (if needed):
```sql
-- For status + location combined queries
CREATE INDEX idx_status_barangay ON missing_pets_reports(status, location_barangay);
-- Use case: "Show all missing pets in Burol"
SELECT * FROM missing_pets_reports WHERE status = 'missing' AND location_barangay = 'Burol';

-- For date-range queries on last_seen_at
CREATE INDEX idx_last_seen ON missing_pets_reports(last_seen_at);
-- Use case: Reports from last 7 days
SELECT * FROM missing_pets_reports WHERE last_seen_at >= DATE_SUB(NOW(), INTERVAL 7 DAY);

-- For user-specific reports
CREATE INDEX idx_user_reports ON missing_pets_reports(user_id);
-- Use case: My reported pets
SELECT * FROM missing_pets_reports WHERE user_id = 1;
```

## Consistency with Existing System: YES

**Reasoning:**

| Aspect | Analysis |
|--------|----------|
| **Naming Convention** | Uses `report_id` as PK, similar to `adoption_applications` (uses `id`) and `appointments` (uses `id`). Uses snake_case throughout. |
| **FK References** | Links to `users(id)` and `pets(pet_id)` - consistent with existing relationships in `pet_owners` table. |
| **Status Enum** | Matches existing `missing_pets.status` enum values (missing/found/reunited). |
| **Photo Handling** | Uses separate `*_photos` table pattern - consistent with `media` table in system. |
| **Timestamps** | Uses `created_at`/`updated_at` timestamps - consistent with all tables. |
| **Soft Deletes** | If needed, can add `deleted_at` - consistent with system (check if users table uses it). |
| **Cascade Delete** | Uses `ON DELETE CASCADE` - consistent with other tables like `pet_owners`. |

**The design is NOT an outlier because:**
1. Follows same FK pattern as `pet_owners` (user_id → users)
2. Uses same timestamp convention as all other tables
3. Status enum matches existing `missing_pets` table
4. Photo separation follows `media` table pattern
5. Indexing strategy consistent with `appointments` table
